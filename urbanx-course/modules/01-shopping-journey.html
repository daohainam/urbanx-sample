Behind the Scenes of a Single Click

You have placed orders before. You clicked "Buy," got a confirmation email, and a package showed up. Simple, right? Under the hood, placing one order silently triggers six different specialist programs working together — each one expert at its single job, none of them aware of the others' internals.

UrbanX is built as a microservices application. Think of it like a documentary crew at a restaurant: front-of-house, kitchen, sommelier, pastry chef — all coordinating seamlessly, each invisible to the guest. This module is that documentary.

Three Details Worth Pausing On

The flow above hides some clever engineering. Here are three moments in the journey that are more interesting than they first appear.

1
The Gateway checks your identity

Every request carries a JWT token issued by the Identity Service. The API Gateway validates this token before forwarding the request. No valid token, no access.

2
Two writes, one transaction

When the Order Service saves your order to PostgreSQL, it also writes to an Outbox table in the very same database transaction. If the server crashes right after, the event is still there — never lost.

3
Services never call each other directly

The Order Service does not phone the Inventory Service. Instead it publishes an event to Kafka. Inventory subscribes and reacts on its own schedule. This loose coupling means services can be updated, restarted, or scaled independently.

Four Patterns Powering the Journey

The shopping flow you just traced is built on four architectural patterns. This course covers each in depth — here is a preview of what each one solves.

📬

Transactional Outbox

Events are saved to the database before being sent, so they are never lost even if the server crashes mid-operation.

🎭

Choreography Saga

Multiple services coordinate without a conductor — each reacts to events and passes the baton to the next specialist.

✂️

CQRS

Reading and writing use separate databases optimized for each job — fast writes in PostgreSQL, fast searches in Elasticsearch.

📡

Event-Driven Architecture

Services talk by broadcasting events, not by calling each other directly — making the whole system resilient and decoupled.
