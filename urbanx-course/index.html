Three Details Worth Pausing On

The flow above hides some clever engineering. Here are three moments in the journey that are more interesting than they first appear.

1
The Gateway checks your identity

Every request carries a JWT token issued by the Identity Service. The API Gateway validates this token before forwarding the request. No valid token, no access.

2
Two writes, one transaction

When the Order Service saves your order to PostgreSQL, it also writes to an Outbox table in the very same database transaction. If the server crashes right after, the event is still there — never lost.

3
Services never call each other directly

The Order Service does not phone the Inventory Service. Instead it publishes an event to Kafka. Inventory subscribes and reacts on its own schedule. This loose coupling means services can be updated, restarted, or scaled independently.

UrbanX is not one big app. It is seven small programs — and that specialization is a deliberate, powerful decision.

One Doctor Does Everything — What Could Go Wrong?

Imagine a hospital where a single doctor handles cardiology, ophthalmology, and surgery simultaneously. Exhausted, error-prone, and impossible to scale. Instead, hospitals hire specialists: each expert in one domain, accountable for one set of outcomes.

UrbanX works the same way. If the Payment Service crashes, your shopping cart still works. If the Catalog Service is overloaded with search queries, orders keep processing. Specialization buys you independence — and independence buys you resilience.

Each service owns its own database. No service can reach into another service's data directly. They only communicate via Kafka events or the HTTP API they expose.

How the Gateway Decides Where to Send Requests

The API Gateway uses YARP (a reverse proxy library) configured via JSON. Each route maps a URL pattern to one specialist service. The Gateway itself never processes business logic — it only forwards.

CODE 
"Routes": {
  "catalog-route": {
    "ClusterId": "catalog",
    "Match": { "Path": "/api/products/{**catch-all}" }
  },
  "order-route": {
    "ClusterId": "order",
    "Match": { "Path": "/api/orders/{**catch-all}" }
  },
  "payment-route": {
    "ClusterId": "payment",
    "Match": { "Path": "/api/payments/{**catch-all}" }
  },
  "identity-route": {
    "ClusterId": "identity",
    "Match": { "Path": "/connect/{**catch-all}" }
  }
}
PLAIN ENGLISH

Any URL starting with /api/products goes to the Catalog Service — the product specialist.

Anything under /api/orders or /api/cart goes to the Order Service — carts and orders live together.

Payment requests under /api/payments go to the Payment Service — the only one that knows about Stripe.

The /connect/ path is for login — it goes to the Identity Service running Duende IdentityServer.

The Gateway itself never processes business logic — it just reads this map and forwards.

🔒
Rate Limiting at the Gate

The Gateway also enforces rate limiting — 100 requests per minute per client. This single rule protects all seven services at once.

How Every Service Checks Your Identity

03

The Messaging Backbone

How seven specialists talk to each other — without ever interrupting each other.

The Office Bulletin Board

Imagine a shared office with seven specialists. The clumsy way to coordinate is to tap everyone on the shoulder — every time, for every update. They get interrupted, they have to drop what they're doing, and if one person is out sick, your whole plan stalls.

UrbanX does something smarter: it uses a message broker called Kafka — the shared bulletin board. Instead of calling Inventory directly, the Order Service pins a note. Inventory reads it when it's ready. Nobody is interrupted. Nobody misses it.

📌
Opening Hook

What if the Inventory Service crashes while you are checking out? In a direct-call system, your order fails instantly. With Kafka, the order message just waits on the board until Inventory comes back — and then it catches up.

Topics: Named Channels on the Board

Every note on the bulletin board goes into a named slot called a topic. UrbanX has five topics — one for each important event type that crosses service boundaries.

order.created Order Service → Inventory Service: triggers stock reservation for new orders
inventory.events Inventory Service → Order Service: carries Reserved or ReservationFailed results
payment.events Payment Service → Order Service: carries Completed or Failed payment results
order.cancelled Order Service → Inventory Service: triggers stock release when payment fails
catalog.products Catalog Service → Catalog Service: syncs product writes to Elasticsearch
💡
Why Should I Care?

Understanding topics means you can tell an AI “add a new service that listens to payment events” and know exactly what that means: subscribe to the payment.events topic. No changes needed to the Payment Service whatsoever.

Reading the Code: The Inventory Consumer

Here is how the Inventory Service listens for new orders. Every line has a purpose — let's decode it together.

CODE 
var config = new ConsumerConfig
{
  BootstrapServers = _kafkaSettings.BootstrapServers,
  GroupId          = "inventory-service",
  AutoOffsetReset  = AutoOffsetReset.Earliest,
  EnableAutoCommit = false
};

using var consumer = new ConsumerBuilder<string, string>(config).Build();
consumer.Subscribe("order.created");

while (!stoppingToken.IsCancellationRequested)
{
  var result     = consumer.Consume(stoppingToken);
  var orderEvent = JsonSerializer.Deserialize<OrderEvent>(result.Message.Value);
  await ProcessOrderAsync(orderEvent, stoppingToken);
  consumer.Commit(result);
}
PLAIN ENGLISH

Connect to the Kafka server and join the consumer group called “inventory-service”.

GroupId means: if multiple Inventory instances run at the same time, they share the work — no message is processed twice.

EnableAutoCommit=false: we manually confirm receipt only after successful processing — no lost messages if we crash mid-way.

Subscribe to the order.created topic — start listening for new orders.

Loop forever: wait for a message to arrive, deserialize it from JSON, process it (reserve stock), then manually commit the offset to confirm it was handled.

The Resilience Guarantee

Kafka stores messages in an ordered log. When the Inventory Service goes offline, messages queue up at the exact offset where it stopped reading. When it restarts, it resumes from that bookmark — no messages skipped, no messages lost.

This is what “decoupled in time” means: the publisher does not wait for the consumer. The consumer can catch up from any point in history.

💡
At-least-once delivery

At-least-once delivery: Kafka guarantees every message is delivered at least once. If the consumer crashes mid-processing, it gets the message again on restart. Services must handle duplicates gracefully — this is called being idempotent. The Transactional Outbox pattern (Module 4) and the Order Saga (Module 5) show exactly how UrbanX achieves this.

🔗
pub-sub in one sentence

Pub-sub means the Payment Service never needs to know that an Analytics Service exists — Analytics simply subscribes to payment.events and the Payment Service is never touched.

04

Never Lose a Message

The Transactional Outbox pattern — why writing to two systems at once is impossible, and how UrbanX solves it.

The Dangerous Moment

Imagine your order gets saved to the database — and then the server crashes before the message reaches Kafka. The order exists, but Inventory never heard about it. Stock is never reserved. The customer expects delivery that will never come.

This is not a hypothetical. In any distributed system, a crash, network timeout, or even a routine deployment can happen between two separate writes. Without a safety net, that window is catastrophic.

1
Order saved to database ✓

The Order record is written to PostgreSQL successfully.

💥
Server crashes here

A power failure, OOM kill, deployment restart — any crash at this exact moment is catastrophic.

3
Kafka message never sent ✗

The code that publishes to Kafka never runs.

4
Order exists, Inventory never knew ✗

The order is confirmed but stock is never reserved. The customer expects delivery that will never come.

⚠️
The Core Problem

Writing to two different systems — a database and a message broker — can never be atomic. There is always a gap. The Transactional Outbox pattern closes that gap.

The Surgeon's Checklist Solution

Before surgery, every step is written on a form — even if the power goes out mid-operation, staff know exactly what was done and what comes next. The Outbox table is that checklist.

Instead of calling Kafka directly, the Order Service writes the event into its own database, in the same transaction as the order itself. A separate background service (the OutboxRelayService) then reads those pending entries and publishes them to Kafka.

1
Single database transaction

Order record + Outbox entry are written together. Both succeed, or neither does. The crash window is eliminated.

2
Background relay polls every 5 seconds

The OutboxRelayService scans for unprocessed entries and publishes each one to Kafka.

3
Mark as processed

After a successful publish, ProcessedAt is set to now. The entry is never published again.

4
Retry on failure

If Kafka is temporarily unavailable, RetryCount increments. The relay will try again on the next poll. Up to 5 attempts.

Code: The Atomic Write

Here is the exact code in the Order Service that makes the crash-safe write. Notice that both the order and the outbox entry are saved in one SaveChangesAsync call.

CODE 
await db.Orders.AddAsync(order);
await db.OutboxMessages.AddAsync(new OutboxMessage
{
  Id          = Guid.NewGuid(),
  EventType   = "OrderCreated",
  Payload     = JsonSerializer.Serialize(orderEvent),
  CreatedAt   = DateTime.UtcNow,
  ProcessedAt = null,
  RetryCount  = 0
});
await db.SaveChangesAsync();
PLAIN ENGLISH

Save the new Order entity to the Orders table.

In the same database transaction, save a new OutboxMessage — a record of the event that needs to be published.

ProcessedAt=null means: not yet published to Kafka.

RetryCount=0: no delivery attempts yet.

SaveChangesAsync commits BOTH the order AND the outbox entry atomically — either both succeed or neither does.

If the server crashes after this line: both records exist in the database. The relay will pick up and publish to Kafka on restart.

Code: The Relay Loop

The OutboxRelayService is a background service that runs forever. Every 5 seconds it scans for unpublished messages and sends them to Kafka.

CODE 
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
  while (!stoppingToken.IsCancellationRequested)
  {
    var pending = await _db.OutboxMessages
      .Where(m => m.ProcessedAt == null && m.RetryCount < 5)
      .Take(50)
      .ToListAsync(stoppingToken);

    foreach (var msg in pending)
    {
      try
      {
        await _producer.ProduceAsync(msg.EventType, msg.Payload);
        msg.ProcessedAt = DateTime.UtcNow;
      }
      catch (Exception ex)
      {
        _logger.LogError(ex, "Failed to publish outbox message {Id}", msg.Id);
        msg.RetryCount++;
      }
    }

    await _db.SaveChangesAsync(stoppingToken);
    await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
  }
}
PLAIN ENGLISH

This background service runs forever until the app stops.

Find messages not yet published (ProcessedAt=null) with fewer than 5 failed attempts — these are the ones still waiting to be sent.

Take up to 50 at a time — process in batches for efficiency.

Publish each one to Kafka, using EventType as the topic name.

Mark as processed by setting ProcessedAt to now — it will never be picked up again.

If publishing fails: increment RetryCount — the message will be retried on the next poll cycle.

Wait 5 seconds, then poll again — repeat forever.

💡
At-least-once, handled gracefully

If the relay publishes to Kafka but crashes before marking ProcessedAt, the message will be published again on restart. The consumer receives a duplicate. That is why consumers must be idempotent — the next code snippet shows exactly how.

Code: The Idempotent Consumer

What happens if Inventory receives the same order.cancelled event twice? Without protection, it would try to release stock that was already released — corrupting your inventory counts. This code makes that safe.

CODE 
var reservations = await db.InventoryReservations
  .Where(r => r.OrderId == orderId
            && r.Status  == ReservationStatus.Reserved)
  .ToListAsync();

if (reservations.Count == 0) return;

foreach (var r in reservations)
{
  r.Status = ReservationStatus.Released;
  item.QuantityReserved -= r.Quantity;
}
PLAIN ENGLISH

Find all reservations for this order that are still in Reserved status — the ones that actually need releasing.

If there are none — they were already released — exit immediately. Do nothing.

This single check makes the whole operation idempotent: processing the same message 10 times has the same result as processing it once.

Only if reservations exist: release each one — set status to Released and reduce the reserved quantity count.

05

The Order Saga

Three databases. One order. If any step fails, the others must undo their work. Normal database transactions cannot do this — so we use a Saga.

Why a normal transaction is not enough

Fulfilling an order requires three things to happen across three different services: reserve inventory, charge payment, and confirm the order. If any one fails, the others must undo their work.

That is not possible with a normal distributed transaction. PostgreSQL cannot reach into Stripe. Inventory cannot lock Payment. Each service owns its own database.

🏃
Think of it as a relay race.

Order passes the baton to Inventory ("reserve the stock"), which passes it to Payment ("charge the card"), which passes it back to Order ("you are confirmed"). If a runner drops the baton, the team runs the race in reverse to undo what was done.

🎭
This pattern is called a Saga pattern.

UrbanX uses choreography — each service reacts to Kafka events on its own. There is no central controller.

The order state machine

An order cannot jump from Confirmed straight to Delivered. This code enforces every legal transition and rejects all others.

CODE 
public static readonly IReadOnlyDictionary<OrderStatus, IReadOnlySet<OrderStatus>> Allowed =
    new Dictionary<OrderStatus, IReadOnlySet<OrderStatus>>
{
    [OrderStatus.Confirmed]      = new HashSet<OrderStatus> { OrderStatus.Preparing,      OrderStatus.Cancelled },
    [OrderStatus.Preparing]      = new HashSet<OrderStatus> { OrderStatus.ReadyForPickup, OrderStatus.Cancelled },
    [OrderStatus.ReadyForPickup] = new HashSet<OrderStatus> { OrderStatus.InTransit,      OrderStatus.Cancelled },
    [OrderStatus.InTransit]      = new HashSet<OrderStatus> { OrderStatus.Delivered,      OrderStatus.Cancelled },
};
PLAIN ENGLISH

This is a whitelist: only these status transitions are allowed.

Confirmed can go to Preparing or Cancelled — nothing else.

You cannot skip from Confirmed directly to Delivered.

Every service update checks this table before changing status.

Invalid transitions throw an error — protecting the order from entering impossible states.

How compensation is triggered in code

When the Order Service receives a payment failure event, it cancels the order and writes an OrderCancelled event to the Outbox — all in a single database transaction.

CODE 
if (paymentEvent.EventType == "Failed")
{
    order.Status = OrderStatus.Cancelled;
    order.AddStatusHistory(OrderStatus.Cancelled, "Payment failed");

    await db.OutboxMessages.AddAsync(new OutboxMessage
    {
        EventType = "OrderCancelled",
        Payload   = JsonSerializer.Serialize(new OrderCancelledEvent
        {
            OrderId    = order.Id,
            OccurredAt = DateTime.UtcNow
        })
    });

    await db.SaveChangesAsync();
}
PLAIN ENGLISH

Payment event type is Failed — the card was declined or an error occurred.

Update the order status to Cancelled.

Record this in the status history: when it happened and why.

Write an OrderCancelled event to the Outbox — in the same transaction as the status update.

The Outbox guarantees Inventory will hear about this cancellation, even if the server restarts now.

Why choreography makes extension easy

In UrbanX, there is no central service that says "Inventory, reserve stock. Payment, charge the card." Instead, each service listens for the right Kafka event and reacts independently. This is choreography — as opposed to orchestration.

1
Add a new service

Say you need to notify a warehouse system when an order is confirmed. Create a new Warehouse Service.

2
Subscribe to the right event

The Warehouse Service subscribes to order.confirmed on Kafka. That topic already exists.

3
Zero changes to existing services

Order Service, Inventory Service, Payment Service — none of them change at all. The saga grows naturally.

💡
Choreography vs. Orchestration

In choreography (UrbanX's approach), each service reacts to events independently — like a jazz ensemble improvising together. In orchestration, a central "order manager" service tells each service what to do — like a conductor. Choreography is more resilient (no single point of failure) but harder to visualize. 06

Two Databases, One Truth

PostgreSQL stores products accurately. Elasticsearch finds them instantly. Kafka keeps them in sync. This is CQRS.

The right database for the right job

PostgreSQL is brilliant at storing products accurately — every field, every constraint, every relationship. But a full-text search for "wireless headphones" across 10,000 products? That is not what relational databases are built for. Elasticsearch is. So UrbanX uses both — each for what it does best.

This pattern is called CQRS. Writes go to the write model (PostgreSQL). Reads come from the read model (Elasticsearch). Kafka bridges the gap.

📚
Think of a library with two card catalogs.

The acquisitions desk uses a detailed ledger (PostgreSQL) — author, ISBN, condition, purchase date. The public search terminal (Elasticsearch) uses a fast index built for "find me books about dragons." When a new book arrives, the ledger is updated first, then the index is refreshed.

⚡
The trade-off you accept consciously.

The search index lags the database by a few seconds — called eventual consistency. For a product catalog, this is completely fine. For a bank balance, it would not be.

What a search query looks like in code

When a customer types "wireless headphones" into the search bar, this query runs against the index (Elasticsearch) — not against PostgreSQL.

CODE 
var response = await _client.SearchAsync<ProductDocument>(s => s
    .Index("catalog-products")
    .Query(q => q
        .Bool(b => b
            .Must(m => m
                .MultiMatch(mm => mm
                    .Fields(f => f
                        .Field(p => p.Name)
                        .Field(p => p.Description))
                    .Query(searchTerm)))
            .Filter(f => f
                .Term(t => t
                    .Field(p => p.IsActive)
                    .Value(true)))))
    .From((page - 1) * pageSize)
    .Size(pageSize));
PLAIN ENGLISH

Search the catalog-products Elasticsearch index (not the PostgreSQL database).

Use a Bool query — like combining conditions in a SQL WHERE clause.

Must: the search term must appear somewhere in this product.

MultiMatch: look in BOTH the name field AND the description — find partial matches too.

Filter: only show products where IsActive=true — hidden products never appear in search.

From/Size: pagination — skip the first N results, return the next batch.

How a write triggers the sync pipeline

Notice what is not in this code: Elasticsearch is never called here. The write side only knows about PostgreSQL. Elasticsearch is updated asynchronously, later, via Kafka. This is the CQRS split in action.

CODE 
app.MapPut("/api/products/{id}", async (Guid id, UpdateProductRequest req, AppDbContext db) =>
{
    var product = await db.Products.FindAsync(id);
    if (product is null) return Results.NotFound();

    product.Name        = req.Name;
    product.Description = req.Description;
    product.Price       = req.Price;
    product.UpdatedAt   = DateTime.UtcNow;

    await db.OutboxMessages.AddAsync(new OutboxMessage
    {
        EventType = "ProductUpdated",
        Payload   = JsonSerializer.Serialize(new ProductEvent(product))
    });

    await db.SaveChangesAsync();
    return Results.Ok(product);
});
PLAIN ENGLISH

Update the product fields in PostgreSQL.

In the same transaction, write a ProductUpdated event to the Outbox.

SaveChangesAsync commits both changes atomically.

The Outbox Relay will pick this up and publish to catalog.products within 5 seconds.

Note: Elasticsearch is NOT updated here — it is updated asynchronously via Kafka.

The Kafka consumer that updates Elasticsearch

On the read side, a Kafka consumer watches the catalog.products topic. Each time a ProductUpdated event arrives, it rebuilds the denormalized search document and pushes it to Elasticsearch.

ProductDocument A flat, search-optimized representation of a product — only the fields needed for search queries. No foreign keys, no joins.
IndexAsync(document) Replaces the existing document in Elasticsearch with the new version. If it does not exist yet, it is created.
consumer.Commit(result) Tells Kafka that this message has been successfully processed. Only called after the index is updated — guarantees at-least-once delivery.
ℹ️
The trade-off you accept with CQRS

A product updated right now might show the old name in search results for up to ~10 seconds. Engineers call this eventual consistency. For a product catalog, this is completely fine. For a bank balance, it would not be. Choosing CQRS means accepting this trade-off consciously — and it is almost always the right call for search features.

💡
The same pattern, used twice

UrbanX uses Kafka for two completely different purposes: the Order Saga (coordinating services) and CQRS (syncing databases). Same infrastructure, two distinct use cases. This is why Kafka is described as the "backbone" of the architecture.
