Seven Specialists

UrbanX is not one big app. It is seven small programs — and that specialization is a deliberate, powerful decision.

One Doctor Does Everything — What Could Go Wrong?

Imagine a hospital where a single doctor handles cardiology, ophthalmology, and surgery simultaneously. Exhausted, error-prone, and impossible to scale. Instead, hospitals hire specialists: each expert in one domain, accountable for one set of outcomes.

UrbanX works the same way. If the Payment Service crashes, your shopping cart still works. If the Catalog Service is overloaded with search queries, orders keep processing. Specialization buys you independence — and independence buys you resilience.

🏥
The Key Insight

Each service owns its own database. No service can reach into another service's data directly. They only communicate via Kafka events or the HTTP API they expose.

How the Gateway Decides Where to Send Requests

The API Gateway uses YARP (a reverse proxy library) configured via JSON. Each route maps a URL pattern to one specialist service. The Gateway itself never processes business logic — it only forwards.

CODE 
"Routes": {
  "catalog-route": {
    "ClusterId": "catalog",
    "Match": { "Path": "/api/products/{**catch-all}" }
  },
  "order-route": {
    "ClusterId": "order",
    "Match": { "Path": "/api/orders/{**catch-all}" }
  },
  "payment-route": {
    "ClusterId": "payment",
    "Match": { "Path": "/api/payments/{**catch-all}" }
  },
  "identity-route": {
    "ClusterId": "identity",
    "Match": { "Path": "/connect/{**catch-all}" }
  }
}
PLAIN ENGLISH

Any URL starting with /api/products goes to the Catalog Service — the product specialist.

Anything under /api/orders or /api/cart goes to the Order Service — carts and orders live together.

Payment requests under /api/payments go to the Payment Service — the only one that knows about Stripe.

The /connect/ path is for login — it goes to the Identity Service running Duende IdentityServer.

The Gateway itself never processes business logic — it just reads this map and forwards.

🔒
Rate Limiting at the Gate

The Gateway also enforces rate limiting — 100 requests per minute per client. This single rule protects all seven services at once.

How Every Service Checks Your Identity

The Identity Service uses OIDC via Duende IdentityServer. After login, every service independently validates the JWT using a cached public key — they do not call Identity on every request.

CODE 
builder.Services
  .AddAuthentication(
    JwtBearerDefaults.AuthenticationScheme)
  .AddJwtBearer(options =>
  {
    options.Authority =
      "http://identity:5005";
    options.Audience =
      "urbanx-api";
    options.RequireHttpsMetadata
      = false;
  });
PLAIN ENGLISH

Every service validates tokens by checking with the Identity Service — they all share this exact configuration.

Authority — the Identity Service URL. Go here to fetch the public key used to verify token signatures.

Audience — only accept tokens issued specifically for this API. Tokens from other systems are rejected.

RequireHttpsMetadata = false — development only. In production, HTTPS would be required for all token traffic.

📜
Authorization Policies

Beyond authentication, services also define authorization policies: CustomerOnly, MerchantOnly, and CustomerOrMerchant. An endpoint decorated with [Authorize(Policy = "MerchantOnly")] will reject a customer's request even if their token is valid.

The Services, In Their Own Words

Imagine the seven specialists in a group chat. Each one introduces itself — what it owns, what it does not touch, and how it fits into the wider system.

ID
Identity Service

I handle logins. Give me a username and password, I will give you a JWT token valid for 60 minutes.

CA
Catalog Service

I manage the product catalogue. Writes go to PostgreSQL, searches go to Elasticsearch. I keep them in sync via Kafka events.

OR
Order Service

Shopping carts and orders live with me. I also run the Saga — coordinating inventory and payment so the whole checkout stays consistent.

IN
Inventory Service

I track stock. When Order tells me a new order was placed, I reserve the items. If payment fails, I release them — that is Saga compensation.

PA
Payment Service

I talk to Stripe. I never share those credentials with anyone else. My job is to charge and report results — nothing more.

ID
Identity Service

And none of us call each other directly. We talk through Kafka — that is next module.

Architecture Decision Practice

Good architecture is about decisions — specifically, which service should own which data. These two questions test that judgement. Think about domain ownership before selecting.

You want to add a "Favorite Products" feature where customers save products. Which service should own this data?
