The Order Saga

Three databases. One order. If any step fails, the others must undo their work. Normal database transactions cannot do this — so we use a Saga.

Why a normal transaction is not enough

Fulfilling an order requires three things to happen across three different services: reserve inventory, charge payment, and confirm the order. If any one fails, the others must undo their work.

That is not possible with a normal distributed transaction. PostgreSQL cannot reach into Stripe. Inventory cannot lock Payment. Each service owns its own database.

🏃
Think of it as a relay race.

Order passes the baton to Inventory ("reserve the stock"), which passes it to Payment ("charge the card"), which passes it back to Order ("you are confirmed"). If a runner drops the baton, the team runs the race in reverse to undo what was done.

🎭
This pattern is called a Saga pattern.

UrbanX uses choreography — each service reacts to Kafka events on its own. There is no central controller.

The order state machine

An order cannot jump from Confirmed straight to Delivered. This code enforces every legal transition and rejects all others.

CODE 
public static readonly IReadOnlyDictionary<OrderStatus, IReadOnlySet<OrderStatus>> Allowed =
    new Dictionary<OrderStatus, IReadOnlySet<OrderStatus>>
{
    [OrderStatus.Confirmed]      = new HashSet<OrderStatus> { OrderStatus.Preparing,      OrderStatus.Cancelled },
    [OrderStatus.Preparing]      = new HashSet<OrderStatus> { OrderStatus.ReadyForPickup, OrderStatus.Cancelled },
    [OrderStatus.ReadyForPickup] = new HashSet<OrderStatus> { OrderStatus.InTransit,      OrderStatus.Cancelled },
    [OrderStatus.InTransit]      = new HashSet<OrderStatus> { OrderStatus.Delivered,      OrderStatus.Cancelled },
};
PLAIN ENGLISH

This is a whitelist: only these status transitions are allowed.

Confirmed can go to Preparing or Cancelled — nothing else.

You cannot skip from Confirmed directly to Delivered.

Every service update checks this table before changing status.

Invalid transitions throw an error — protecting the order from entering impossible states.

The failure path — card declined

When payment fails, the saga does not just stop — it runs backwards. This is called compensation. Watch how the system gracefully unwinds.

OR
Order Service

Card submitted. Waiting for Payment Service result...

PY
Payment Service

Called Stripe. Card declined: insufficient funds. Publishing PaymentFailed to payment.events.

OR
Order Service

PaymentFailed received. Setting order status to Cancelled. Publishing OrderCancelled to order.cancelled.

IN
Inventory Service

OrderCancelled received for order #abc123. Finding Reserved reservations...

IN
Inventory Service

Found 2x Wireless Headphones reserved. Releasing. QuantityReserved back to 0.

IN
Inventory Service

Done. Stock is available again for other customers.

OR
Order Service

Compensation complete. The system is back to exactly the state it was before the order. No stock locked, no charge made.

How compensation is triggered in code

When the Order Service receives a payment failure event, it cancels the order and writes an OrderCancelled event to the Outbox — all in a single database transaction.

CODE 
if (paymentEvent.EventType == "Failed")
{
    order.Status = OrderStatus.Cancelled;
    order.AddStatusHistory(OrderStatus.Cancelled, "Payment failed");

    await db.OutboxMessages.AddAsync(new OutboxMessage
    {
        EventType = "OrderCancelled",
        Payload   = JsonSerializer.Serialize(new OrderCancelledEvent
        {
            OrderId    = order.Id,
            OccurredAt = DateTime.UtcNow
        })
    });

    await db.SaveChangesAsync();
}
PLAIN ENGLISH

Payment event type is Failed — the card was declined or an error occurred.

Update the order status to Cancelled.

Record this in the status history: when it happened and why.

Write an OrderCancelled event to the Outbox — in the same transaction as the status update.

The Outbox guarantees Inventory will hear about this cancellation, even if the server restarts now.

Why choreography makes extension easy

In UrbanX, there is no central service that says "Inventory, reserve stock. Payment, charge the card." Instead, each service listens for the right Kafka event and reacts independently. This is choreography — as opposed to orchestration.

1
Add a new service

Say you need to notify a warehouse system when an order is confirmed. Create a new Warehouse Service.

2
Subscribe to the right event

The Warehouse Service subscribes to order.confirmed on Kafka. That topic already exists.

3
Zero changes to existing services

Order Service, Inventory Service, Payment Service — none of them change at all. The saga grows naturally.

💡
Choreography vs. Orchestration

In choreography (UrbanX's approach), each service reacts to events independently — like a jazz ensemble improvising together. In orchestration, a central "order manager" service tells each service what to do — like a conductor. Choreography is more resilient (no single point of failure) but harder to visualize.
