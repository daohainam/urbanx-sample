04

Never Lose a Message

The Transactional Outbox pattern — why writing to two systems at once is impossible, and how UrbanX solves it.

The Dangerous Moment

Imagine your order gets saved to the database — and then the server crashes before the message reaches Kafka. The order exists, but Inventory never heard about it. Stock is never reserved. The customer expects delivery that will never come.

This is not a hypothetical. In any distributed system, a crash, network timeout, or even a routine deployment can happen between two separate writes. Without a safety net, that window is catastrophic.

1
Order saved to database ✓

The Order record is written to PostgreSQL successfully.

💥
Server crashes here

A power failure, OOM kill, deployment restart — any crash at this exact moment is catastrophic.

3
Kafka message never sent ✗

The code that publishes to Kafka never runs.

4
Order exists, Inventory never knew ✗

The order is confirmed but stock is never reserved. The customer expects delivery that will never come.

⚠️
The Core Problem

Writing to two different systems — a database and a message broker — can never be atomic. There is always a gap. The Transactional Outbox pattern closes that gap.

The Surgeon's Checklist Solution

Before surgery, every step is written on a form — even if the power goes out mid-operation, staff know exactly what was done and what comes next. The Outbox table is that checklist.

Instead of calling Kafka directly, the Order Service writes the event into its own database, in the same transaction as the order itself. A separate background service (the OutboxRelayService) then reads those pending entries and publishes them to Kafka.

1
Single database transaction

Order record + Outbox entry are written together. Both succeed, or neither does. The crash window is eliminated.

2
Background relay polls every 5 seconds

The OutboxRelayService scans for unprocessed entries and publishes each one to Kafka.

3
Mark as processed

After a successful publish, ProcessedAt is set to now. The entry is never published again.

4
Retry on failure

If Kafka is temporarily unavailable, RetryCount increments. The relay will try again on the next poll. Up to 5 attempts.

Code: The Atomic Write

Here is the exact code in the Order Service that makes the crash-safe write. Notice that both the order and the outbox entry are saved in one SaveChangesAsync call.

CODE 
await db.Orders.AddAsync(order);
await db.OutboxMessages.AddAsync(new OutboxMessage
{
  Id          = Guid.NewGuid(),
  EventType   = "OrderCreated",
  Payload     = JsonSerializer.Serialize(orderEvent),
  CreatedAt   = DateTime.UtcNow,
  ProcessedAt = null,
  RetryCount  = 0
});
await db.SaveChangesAsync();
PLAIN ENGLISH

Save the new Order entity to the Orders table.

In the same database transaction, save a new OutboxMessage — a record of the event that needs to be published.

ProcessedAt=null means: not yet published to Kafka.

RetryCount=0: no delivery attempts yet.

SaveChangesAsync commits BOTH the order AND the outbox entry atomically — either both succeed or neither does.

If the server crashes after this line: both records exist in the database. The relay will pick up and publish to Kafka on restart.

Code: The Relay Loop

The OutboxRelayService is a background service that runs forever. Every 5 seconds it scans for unpublished messages and sends them to Kafka.

CODE 
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
  while (!stoppingToken.IsCancellationRequested)
  {
    var pending = await _db.OutboxMessages
      .Where(m => m.ProcessedAt == null && m.RetryCount < 5)
      .Take(50)
      .ToListAsync(stoppingToken);

    foreach (var msg in pending)
    {
      try
      {
        await _producer.ProduceAsync(msg.EventType, msg.Payload);
        msg.ProcessedAt = DateTime.UtcNow;
      }
      catch (Exception ex)
      {
        _logger.LogError(ex, "Failed to publish outbox message {Id}", msg.Id);
        msg.RetryCount++;
      }
    }

    await _db.SaveChangesAsync(stoppingToken);
    await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
  }
}
PLAIN ENGLISH

This background service runs forever until the app stops.

Find messages not yet published (ProcessedAt=null) with fewer than 5 failed attempts — these are the ones still waiting to be sent.

Take up to 50 at a time — process in batches for efficiency.

Publish each one to Kafka, using EventType as the topic name.

Mark as processed by setting ProcessedAt to now — it will never be picked up again.

If publishing fails: increment RetryCount — the message will be retried on the next poll cycle.

Wait 5 seconds, then poll again — repeat forever.

💡
At-least-once, handled gracefully

If the relay publishes to Kafka but crashes before marking ProcessedAt, the message will be published again on restart. The consumer receives a duplicate. That is why consumers must be idempotent — the next code snippet shows exactly how.

Code: The Idempotent Consumer

What happens if Inventory receives the same order.cancelled event twice? Without protection, it would try to release stock that was already released — corrupting your inventory counts. This code makes that safe.

CODE 
var reservations = await db.InventoryReservations
  .Where(r => r.OrderId == orderId
            && r.Status  == ReservationStatus.Reserved)
  .ToListAsync();

if (reservations.Count == 0) return;

foreach (var r in reservations)
{
  r.Status = ReservationStatus.Released;
  item.QuantityReserved -= r.Quantity;
}
PLAIN ENGLISH

Find all reservations for this order that are still in Reserved status — the ones that actually need releasing.

If there are none — they were already released — exit immediately. Do nothing.

This single check makes the whole operation idempotent: processing the same message 10 times has the same result as processing it once.

Only if reservations exist: release each one — set status to Released and reduce the reserved quantity count.
